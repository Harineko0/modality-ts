# Multiplatform Native Checker And Local Install

## Goal

Fix the two regressions introduced by the Rust-backed checker packaging:

1. Publish `modality-ts` with native checker binaries for Linux, macOS, and Windows instead of publishing only the Ubuntu-built `.node` file.
2. Make the recommended installation method local dev dependency installation (`npm i -D modality-ts` / package-manager equivalents) instead of global installation, because property files such as `app.props.mjs` import `modality-ts/core` from the target app dependency graph.

The end state should let these user flows work:

- macOS user installs `modality-ts` from npm and runs `npx modality check` or `pnpm exec modality check` without trying to load `modality-checker.linux-x64-gnu.node`.
- Linux CI installs `modality-ts` from npm and runs `modality check`.
- Windows user installs `modality-ts` from npm and runs `modality check`.
- Example/target apps that import `modality-ts/core` have `modality-ts` in `devDependencies`.

## Non-goals

- Do not rewrite the Rust checker or change checker semantics.
- Do not split the package into optional per-platform npm packages unless the single-package artifact matrix proves unworkable. The minimal first fix is one npm package containing multiple platform `.node` files.
- Do not require users to install Rust, Cargo, or `@napi-rs/cli` when consuming `modality-ts` from npm.
- Do not remove the `modality` bin or existing package export paths.
- Do not convert examples into a workspace or alter their app dependencies beyond adding `modality-ts` as a dev dependency.
- Do not publish generated `dist/`, `native/`, or lockfile changes from local development unless the release/publish workflow explicitly generates them.

## Current-State Findings

- `package.json` publishes `"native/**"` and runs `"build:rust": "napi build --platform --release --cargo-cwd crates/checker native"`.
- `.github/workflows/publish.yml` has one `publish` job on `ubuntu-latest`; it runs `pnpm build` and `pnpm publish:npm`, so the npm tarball can contain only the Linux native binary produced on Ubuntu.
- `src/check/native.ts` searches `native/` directories and returns the first file matching `modality-checker.*.node`. On macOS, if the package contains only `modality-checker.linux-x64-gnu.node`, it returns that Linux file and `dlopen` reports an invalid Mach-O slice.
- Local `native/` currently contains `modality-checker.darwin-arm64.node` plus `modality-checker.node -> modality-checker.darwin-arm64.node`, which shows the current local build pattern.
- `native/index.js` is generated by NAPI-RS and contains useful platform/libc triple logic, but the package is ESM (`"type": "module"`), and `src/check/native.ts` currently bypasses that generated CommonJS loader and requires the `.node` file directly.
- `crates/checker/Cargo.toml` builds a `cdylib` N-API addon named `modality_checker`.
- `README.md` currently recommends `npm install -g modality-ts`.
- `examples/demo-app/app.props.mjs`, `examples/checkout-app/app.props.mjs`, and `examples/todo-app/app.props.mjs` import `modality-ts/core`.
- `examples/todo-app/package.json` already has `"devDependencies": { "modality-ts": "^0.0.17" }` and an untracked `examples/todo-app/pnpm-lock.yaml` exists in the current worktree. Preserve user changes there.
- `examples/demo-app/package.json` and `examples/checkout-app/package.json` do not currently declare `modality-ts`.
- `tools/examples-ci.ts` invokes CLI commands directly from source, so it does not currently prove that example app package manifests can resolve `modality-ts/core` after installation.
- GitHub-hosted runner labels currently include Linux x64, Windows x64, macOS Intel, and macOS arm64 labels. Prefer current labels such as `ubuntu-latest`, `windows-latest`, `macos-15-intel`/`macos-26-intel`, and `macos-latest`/`macos-15`/`macos-26`; verify availability against GitHub's hosted runner reference before finalizing the workflow.

## Existing Patterns To Follow

- Keep the Rust checker behind `src/check/native.ts`; TypeScript callers should continue using `checkModel`, `modelInitialStates`, and `modelSuccessors` through `src/check/check-model.ts` and `src/check/model-api.ts`.
- Keep package exports stable. `modality-ts/core`, `modality-ts/check`, and `modality-ts/cli/*` import paths must keep resolving.
- Keep CLI usage examples concise and command-first, matching current README style.
- Keep tests in `test/<area>/*.test.ts` with Vitest.
- Use deterministic file selection in native resolution. Prefer exact platform triples over directory-order-dependent `readdirSync(...).find(...)`.
- Keep workflow jobs explicit. Native packaging should fail if an expected artifact is missing rather than silently publishing a partial package.

## Atomic Implementation Steps

1. Make native binary resolution platform-aware.

   Edit `src/check/native.ts`:
   - Replace directory-order selection of the first `modality-checker.*.node` with explicit runtime candidate names.
   - Add pure helpers that are easy to test:
     - `nativeTriplesForRuntime(platform, arch, libcKind)` returns ordered triple suffixes, for example:
       - macOS arm64: `["darwin-arm64", "darwin-universal"]`
       - macOS x64: `["darwin-x64", "darwin-universal"]`
       - Linux x64 glibc: `["linux-x64-gnu"]`
       - Linux x64 musl: `["linux-x64-musl", "linux-x64-gnu"]` only if intentional fallback is acceptable; otherwise no glibc fallback on musl
       - Linux arm64 glibc: `["linux-arm64-gnu"]` if included in the release matrix
       - Windows x64: `["win32-x64-msvc"]`
     - `detectLinuxLibc()` based on `process.report.getReport().header.glibcVersionRuntime`, with a conservative `ldd` fallback copied from the NAPI-RS generated loader if needed.
     - `resolveNativeBinaryInDirs(nativeDirs, runtime)` that checks `modality-checker.${triple}.node` in candidate order.
   - Keep `modality-checker.node` support for local/development builds, but use it only after exact candidates are missing. Include the expected triples and searched directories in the missing-binary error.
   - Do not require `native/index.js`; it is CommonJS generated output inside an ESM package and is not currently part of the TypeScript source boundary.
   - Stop and report if changing helper visibility would create a public export from `modality-ts/check`; prefer test-only internal exports or colocated tests over public API changes.

2. Add resolver regression tests.

   Add `test/checker/native-resolution.test.ts`:
   - Use temp directories with empty marker files named like native artifacts. Test only resolver paths, not `require()` of dummy `.node` files.
   - Cover:
     - macOS arm64 chooses `modality-checker.darwin-arm64.node` over a Linux file when both exist.
     - macOS arm64 does not choose `modality-checker.linux-x64-gnu.node` when it is the only suffixed file.
     - Linux x64 glibc chooses `modality-checker.linux-x64-gnu.node`.
     - Windows x64 chooses `modality-checker.win32-x64-msvc.node`.
     - `modality-checker.node` is accepted as a last fallback for local current-platform builds.
   - Keep the existing `test/checker/checker.test.ts` smoke test unchanged unless helper exports require a small import adjustment.

3. Define the supported native artifact set.

   Edit `package.json`:
   - Keep `"native/**"` in `files`.
   - Keep local development `build:rust` behavior as a current-platform build unless release packaging needs a separate script.
   - Add release-oriented scripts only if they simplify workflow commands, for example:
     - `"build:rust:target": "napi build --platform --release --cargo-cwd crates/checker native --target"`
     - or avoid this if passing the target directly in workflow YAML is clearer.
   - Decide the initial required artifacts:
     - Required minimum: `modality-checker.linux-x64-gnu.node`, `modality-checker.darwin-arm64.node`, `modality-checker.darwin-x64.node`, `modality-checker.win32-x64-msvc.node`.
     - Strongly consider adding `modality-checker.linux-arm64-gnu.node` because GitHub hosted arm64 Ubuntu labels exist.
     - Add musl (`linux-x64-musl`) only if Alpine support is a release goal now; otherwise make the unsupported-musl error explicit.
   - Do not add optional native package names unless switching to an optional-dependency packaging model.

4. Change the publish workflow to assemble all native artifacts before npm publish.

   Edit `.github/workflows/publish.yml`:
   - Split the current single job into at least:
     - `build-js`: installs dependencies and runs `pnpm typecheck` or `pnpm build` with a current-platform native build if needed for generated types/tests.
     - `build-native`: matrix job that builds one native artifact per target and uploads it.
     - `publish`: downloads all native artifacts into `native/`, verifies the expected filenames exist, builds TypeScript output if needed, runs `npm pack --dry-run --json`, and publishes.
   - Use `pnpm/action-setup@v4`, `actions/setup-node@v4`, and `actions-rs/toolchain` or plain `rustup target add` consistently.
   - Use `pnpm exec napi build --platform --release --cargo-cwd crates/checker native --target <rust-target>` in each matrix job. `pnpm exec napi build --help` confirms this CLI supports `--target`.
   - Suggested target matrix:
     - `ubuntu-latest`, `x86_64-unknown-linux-gnu`, output `native/modality-checker.linux-x64-gnu.node`
     - `ubuntu-24.04-arm` if available for this repository, `aarch64-unknown-linux-gnu`, output `native/modality-checker.linux-arm64-gnu.node`
     - `macos-latest` or another arm64 macOS label, `aarch64-apple-darwin`, output `native/modality-checker.darwin-arm64.node`
     - `macos-15-intel` or `macos-26-intel`, `x86_64-apple-darwin`, output `native/modality-checker.darwin-x64.node`

     - `rm -rf native`

     - run `npm pack --dry-run --json` and confirm the dry-run file list includes `dist/**` and every required `native/modality-checker.*.node`

## Risks, Ambiguities, And Stop Conditions

- GitHub runner labels change over time. Before finalizing `.github/workflows/publish.yml`, verify current macOS Intel and macOS arm64 labels against GitHub's hosted runner reference. Stop and report if an Intel macOS runner is not available for this repository, because dropping `darwin-x64` would be a support decision.
- macOS arm64 and macOS Intel builds may require different runner labels than `macos-latest`; do not rely on `macos-latest` for both architectures.
- Linux musl/Alpine support is ambiguous. If users need Alpine, add `linux-x64-musl` to the required artifact set and workflow matrix; otherwise make the unsupported-platform error clear.
- Windows arm64 support is not part of the minimum unless maintainers request it. Do not expand the matrix endlessly without a support decision.
- NAPI-RS can also publish optional per-platform packages. If the implementation agent chooses that path, stop and produce a revised packaging plan first because it changes npm package names, `optionalDependencies`, publishing order, and failure modes.
- The existing untracked `examples/todo-app/pnpm-lock.yaml` may be a user change. Do not delete or commit it without explicit direction.
- If `npm pack --dry-run --json` does not include downloaded native artifacts in the final publish job, stop before publishing.
- If `pnpm build` in the final publish job deletes or overwrites downloaded cross-platform native artifacts, split TypeScript build and Rust build so the final package assembly preserves all matrix artifacts.
- If resolver tests require exporting internals from `modality-ts/check`, stop and prefer colocated test-only helpers or private named exports that are not added to package exports.
