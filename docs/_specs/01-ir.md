# Spec 01 — The Transition-System IR

Status: draft for review. Companion to `docs/design.md §3`.

The IR is the single contract between the four subsystems: the **extractor** produces it, the **checker** executes it, the **exporter** translates it (TLA+/SMV), and the **replay generator** consumes traces over it. Its design is dominated by one tension: effects must be *executable* (the checker runs them as JS) and, as far as possible, *analyzable* (slicing, export, and partial-order reduction need read/write sets and structured semantics). The IR therefore has a **structured core** and a **declared-footprint opaque escape hatch**.

## 1. Abstract domains

Every state variable has an `AbstractDomain` describing its finite value set. Domains are data (serializable), not code.

```ts
type NumericOverflowPolicy = 'forbid' | 'wrap' | 'saturate';

type AbstractDomain =
  | { kind: 'bool' }
  | { kind: 'enum'; values: readonly string[] }            // literal unions, tags
  | { kind: 'boundedInt'; min: number; max: number;        // finite integer range
      overflow?: NumericOverflowPolicy }
  | { kind: 'intSet'; values: readonly number[];           // sparse finite integer set
      overflow?: NumericOverflowPolicy }
  | { kind: 'option'; inner: AbstractDomain }               // null | value
  | { kind: 'record'; fields: Record<string, AbstractDomain> }
  | { kind: 'tagged';                                       // discriminated union
      tag: string;                                          // discriminant field name
      variants: Record<string, AbstractDomain /* record */> }
  | { kind: 'tokens'; count: number; names?: string[] }     // opaque values distinguished
                                                            // only by identity: data₁, data₂
  | { kind: 'lengthCat' }                                   // collections: '0' | '1' | 'many'
  | { kind: 'boundedList'; inner: AbstractDomain; maxLen: number }; // history stack etc.
```

Notes and rationale:

- **`tokens`** is the workhorse for server data: properties rarely care what the data *is*, only whether it changed, is present, or belongs to the wrong identity (stale-cache bugs need ≥2 tokens: `userA`-data vs `userB`-data). `count` is a bound; exhausting tokens during exploration is a reported bound-hit, not silent reuse.
- **`lengthCat`** is the default array abstraction. Element-sensitive list reasoning uses `boundedList` and must be requested in the overlay — it is the most expensive domain.
- **Finite numeric domains (`boundedInt`, `intSet`).** Numbers *can* be concrete finite state when their domain is statically provable (the SPIN spirit). `boundedInt{min,max}` is a dense range; `intSet{values}` is a sparse set, so a literal union `0 | 2` becomes `intSet{0,2}` and does **not** widen to `0..2` (the domain fingerprint distinguishes the two). The optional `overflow` policy (`forbid` default / `wrap` / `saturate`) is metadata for assignment: reachable overflow is a *checking behaviour*, not erased by static validation (Spec 03). Sources: TS numeric literal unions, branded aliases (`Bounded<Min,Max>`, `Wrapping<Min,Max>`, `Uint8`/`Byte`, `Uint16`, `Short`), and statically readable `zod`/`arktype` integer schemas (Spec 02 §3). `boundedInt` is also used for tool-internal counters (pending ops).
- **Bare `number` and unbounded `string` stay abstract by design.** A non-literal `number`, a float, or an unprovable numeric constraint maps to `tokens` (default `count: 1` = "some value") plus an extraction caveat — never a guessed range (a wrong numeric bound would be unsound). Refinable via an overlay-declared predicate abstraction (a derived `enum`, see Spec 02 §3, §5).

Each domain has derived helpers generated alongside it: `enumerate(): Value[]` (for havoc), `validate(v): boolean` (checker debug mode), and a **concretization witness** requirement used by replay (Spec 04 §2).

## 2. State vector

```ts
interface StateVarDecl {
  id: string;                  // stable, e.g. 'atom:sessionAtom', 'local:CheckoutPage.step',
                               //              'swr:GET /api/user', 'sys:route', 'sys:pending'
  domain: AbstractDomain;
  origin: SourceAnchor | 'system' | 'library-template';
  scope: { kind: 'global' } | { kind: 'route-local'; route: string }
        | { kind: 'mount-local'; id: string; when: ExprIR };
  initial: Value | Value[];    // multiple ⇒ nondeterministic initial value
}
```

A **state** is a record from var id → value. System variables present in every model:

- `sys:route: enum(routePatterns)` and `sys:history: boundedList(route, H)` — current leaf route and back-stack (bound `H` from config). `sys:route` remains the compatibility surface for route-local scopes and properties that reason about the active URL pattern.
- `sys:next:slot:<key>: enum(...)` — optional route-tree slot state for framework adapters (Next.js parallel routes). Values name active route-tree node ids or a distinguished `__none` sentinel.
- `sys:next:phase:<boundaryId>: enum('ready','loading','error','not-found','forbidden','unauthorized')` — finite loading/error/cache phases along a route boundary (not byte-level streaming).
- `sys:next:cache:<key>: enum('empty','fresh','stale','refreshing','error')` — finite cache lifecycle for statically discoverable revalidation tags/paths.
- `sys:pending: boundedList(PendingOp, K)` — in-flight async operations, bound `K` from config. A `PendingOp` is `{ opId: enum, continuation: enum, args: record }` (continuation ids name extracted post-`await` summaries, Spec 02 §7).
- `sys:timer:<id>: enum('idle','scheduled','fired')` — cancellable timer handles synthesized per `setTimeout`/`setInterval` registration site (Spec 02 §11).
- `sys:suspense:<boundaryId>: enum('ready','suspended')` — Suspense boundary state; subtree transitions are gated until `resolve` (Spec 02 §11).
- `local:<Component>.isPending#<n>: bool` and `local:<Component>.deferred:<var>` — concurrent-rendering bookkeeping from `useTransition` / `useDeferredValue` (Spec 02 §11).
- Route-local and mount-local variables exist in every state but hold the distinguished value `⊥` (unmounted) when their mount predicate is false. Mount initializes them; unmount resets to `⊥`. This makes "local state resets on remount" a semantics of the IR, not a convention each effect must remember. `⊥` canonicalizes identically regardless of pre-unmount value — that *is* the reset.
- **`route-local`** remains supported and is equivalent to **`mount-local`** with `when = eq(read("sys:route"), lit(route))`.
- **`mount-local`** scopes a var with an explicit boolean `when: ExprIR` predicate. A var is active exactly when `when` evaluates true. On transition/stabilization, if a local var becomes inactive it is set to `⊥`; if it becomes active from `⊥` it resets to its initial value(s); if it remains active its value is preserved; if it remains inactive it stays `⊥`. Mount guards must not read the scoped var itself (validated at IR check time).

The full state type is code-generated as a TS type (`ModelState`) so that overlay effects and property predicates are type-checked against it.

## 3. Transitions

```ts
interface Transition {
  id: string;                          // stable across regeneration (Spec 02 §9)
  cls: 'user' | 'nav' | 'env' | 'internal' | 'library';
  label: EventLabel;                   // human/replay-facing description, see §6
  source: SourceAnchor[];
  guard: GuardIR;                      // structured predicate over state (no opaque guards)
  effect: EffectIR;
  reads: string[]; writes: string[];   // var ids; MUST over-approximate (checked, §7)
  confidence: 'exact' | 'over-approx' | 'manual';
  phase?: number;                      // commit tier for internal transitions; lower runs first (Spec 03 §3)
}
```

### 3.1 Structured effect language (the analyzable core)

```ts
type EffectIR =
  | { kind: 'assign'; var: string; expr: ExprIR }
  | { kind: 'havoc'; var: string }                          // any value in domain
  | { kind: 'choose'; var: string; among: ExprIR[] }        // bounded nondeterminism
  | { kind: 'if'; cond: ExprIR; then: EffectIR; else: EffectIR }
  | { kind: 'seq'; effects: EffectIR[] }
  | { kind: 'enqueue'; op: string; continuation: string; args: Record<string, ExprIR> }
  | { kind: 'dequeue'; index: number }                      // remove resolved pending op
  | { kind: 'navigate'; mode: 'push' | 'replace' | 'back'; to?: ExprIR }
  | { kind: 'opaque'; ref: OpaqueRef };                     // §3.2

type ExprIR =                                                // total, side-effect-free
  | { kind: 'lit'; value: Value }
  | { kind: 'read'; var: string; path?: string[] }
  | { kind: 'readPre'; var: string; path?: string[] }       // pre-commit snapshot read (batching)
  | { kind: 'readOpArg'; key: string }                      // enqueue-time snapshot in pending op args
  | { kind: 'eq' | 'neq' | 'and' | 'or' | 'not' | 'cond'; args: ExprIR[] }
  | { kind: 'updateField'; target: ExprIR; path: string[]; value: ExprIR } // {...s, f: v}
  | { kind: 'tagIs'; arg: ExprIR; tag: string }
  | { kind: 'lenCat'; arg: ExprIR }
  | { kind: 'freshToken'; domainOf: string }                // next unused token, or bound-hit
  | { kind: 'transitionEnabled'; transitionId: string }     // enabled(t): guard ∧ mount (§ Spec 03 §5)
  | { kind: 'lt' | 'lte' | 'gt' | 'gte'; args: [ExprIR, ExprIR] }  // numeric comparison
  | { kind: 'add' | 'sub' | 'mod'; args: [ExprIR, ExprIR] };       // integer arithmetic (overflow per domain)
```

Numeric comparison/arithmetic nodes operate over `boundedInt`/`intSet` state; `add`/`sub`/`mod` apply the target domain's `overflow` policy at assignment time (Spec 03). `transitionEnabled` is the IR backing for the `enabled(t)` property accessor — exact, because guards are structured.

`read` paths may include **bounded-list indices** (`['0', 'expired']`); extraction unrolls bounded-list comprehensions (`find`/`some`/`every` with M0 lambdas, Spec 02 §6) into `cond`-chains over such indexed reads, so list reasoning needs no new node kinds and the IR stays closed (the architecture's plugins-contribute-instances-not-semantics rule, Spec 05 §3).

`dequeue` is a structured queue effect for async continuations: resolve transitions guard on a pending op, dequeue the resolved index, then apply the post-`await` summary. It is part of the closed IR so the checker, exporter, replay generator, and conformance tooling agree on pending-operation semantics.

`GuardIR = ExprIR` (boolean-valued). Design rule: **guards are always structured** — an opaque guard would make enabledness unanalyzable, which breaks vacuity reporting, export, and any future POR. If extraction can't express a guard structurally, the transition's guard becomes `true` and the unsupported condition moves into the effect as an `if` whose else-branch is identity (over-approximation: the transition may fire as a stutter; sound, and reported).

### 3.2 Opaque effects (the escape hatch)

```ts
interface OpaqueRef {
  module: string; export: string;       // resolved and imported by the checker at load time
  declaredReads: string[]; declaredWrites: string[];
}
// the referenced value: (s: Readonly<ModelState>) => ModelState | ModelState[]
```

Opaque effects come only from the overlay file (manual annotations). Consequences, by subsystem:

- **Checker**: executes them directly; in debug mode validates outputs against domains and against `declaredWrites` (writing an undeclared var is a hard modeling error — it would silently break slicing).
- **Slicing**: uses `declaredReads/Writes`.
- **Export**: a model containing opaque effects exports with those transitions replaced by `havoc(declaredWrites)` — the export is then an over-approximation of the model, stated in the export header. Differential testing (Spec 03 §9) uses structured-only models.
- **Replay**: unaffected (replay drives the real app, not the model).

This split is the load-bearing compromise of the whole IR: full analyzability for what extraction handles, full expressiveness for what humans write, and an explicit, machine-checked footprint contract joining them.

## 4. Nondeterminism model

Three sources, all explicit:

1. **Scheduling**: at each state, every enabled transition is a possible next step (interleaving semantics, one atomic transition at a time).
2. **Environment outcomes**: for each pending op `p`, a family of `env` transitions `resolve(p, outcome)` exists, one per outcome in the op's outcome domain (e.g., `success(token) | error`), filtered by developer `assume()` constraints. Outcome domains are declared per op-id (extracted from the API call's TS return type, abstracted; or overlay-declared).
3. **Effect-internal**: `havoc` / `choose` / multiple-result opaque effects.

There are no probabilistic or prioritized choices in v1. Fairness is not modeled (no unbounded liveness in v1; `leadsToWithin` quantifies over *all* paths within a budget instead — Spec 03 §6).

## 5. Step semantics: macro-steps and run-to-completion

`useEffect` reactions (auth redirects, derived resets) are `internal` transitions with a `triggeredBy: string[]` field (the dependency vars). They do not interleave freely — React runs effects promptly after commit, and modeling them as freely-scheduled steps would create absurd interleavings (user clicks between a state change and its guard redirect at model granularity far below reality).

**Macro-step semantics**: an observable step = one `user`/`nav`/`env`/`library` transition followed by **stabilization** — repeatedly fire any `internal` transition whose `triggeredBy` vars changed since it last ran and whose guard holds, until none is enabled.

- Stabilization order: deterministic (transition id order) *unless* two enabled internal transitions have intersecting write sets, in which case **both orders are explored** (nondeterminism — we must not invent an ordering React doesn't promise across components).
- Divergence: stabilization is capped at `maxInternalSteps` (default 16); exceeding it is a **modeling error** ("effect chain does not stabilize"), not a verification result. This mirrors real infinite-rerender bugs, and the trace to the divergence is itself a useful artifact.
- Only stabilized states enter the state graph and are visible to properties. Verbose traces can show the micro-steps.

This is the standard run-to-completion construction (statecharts, synchronous languages); its known cost is that properties cannot observe mid-stabilization states — accepted, since React batches commits similarly.

## 6. Event labels (the replay contract)

Every non-internal transition carries a label sufficient to *drive the real app*:

```ts
type EventLabel =
  | { kind: 'click' | 'submit'; locator: Locator }              // testid / role+name
  | { kind: 'input'; locator: Locator; valueClass: string }     // abstract input class
  | { kind: 'navigate'; mode: 'push' | 'back'; to?: string }
  | { kind: 'resolve'; op: string; outcome: string }            // network completion
  | { kind: 'focus-revalidate' | 'timer'; key?: string };
```

Locators are extracted from JSX (`data-testid`, accessible role/name) or flagged for overlay completion. For **indexed event families** (handlers rendered inside `xs.map(...)`, Spec 02 §4) the locator is *positional*: a base locator plus the family index (the *n*-th payment-method radio), deterministic at replay because the witness factory fixes list order (Spec 04 §3). A transition without a resolvable locator is still checkable but marked `replayable: false` — counterexamples through it degrade to abstract traces, and the report says so.

## 7. Well-formedness (validated at load time)

1. Every `assign/choose/havoc` target exists and the expression's type matches the domain (structural type-check of ExprIR against domains).
2. `reads/writes` over-approximate the structured effect's actual footprint (computed by walking the EffectIR — must be ⊇; for opaque effects, runtime-validated in debug mode).
3. Guards read only declared `reads`.
4. Route-local vars are written only by transitions of their route or by mount/unmount machinery.
5. All transitions out of `⊥`-valued vars' routes are disabled (can't click on an unmounted component).
6. Every enum/tagged value referenced in expressions exists in the domain.
7. Initial states are valid and stabilized.

## 8. Serialization

- JSON for everything except opaque refs, which serialize as `{module, export}` and resolve at load (the checker imports the overlay module). The serialized model + the overlay module pair is the unit of versioning; the model file embeds a hash of the source files it was extracted from (staleness detection, design §6).
- Canonical JSON ordering so model diffs are reviewable in PRs — the model diff *is* the review artifact for "did this refactor change behavior."

## 9. Library templates

SWR (and later other libraries) ship as **template factories**: functions `(key, options) => { vars: StateVarDecl[], transitions: Transition[] }` written by hand against this IR, instantiated per extracted call site. The SWR v1 template models, per key class: `data: ⊥ | P` where the **payload domain `P` is `D(fetcher return type)`** (Spec 02 §3) — a fetcher returning `Todo[]` yields `data ∈ {⊥, '0', '1', 'many'}`, so properties can distinguish loaded-empty from loaded-some; plus `isValidating: bool`, `error: option`, mount/key-change-triggered fetch with the dedup window as a boolean, `revalidateOnFocus` (env event, on/off per options), `mutate(key)` API effects, and **stale-data retention on error** (a failed revalidation sets `error` but keeps `data` — real SWR behavior, and a state combination property authors forget exists).

Templates additionally export **hook-view helpers** — e.g. `swrView(s, key)` returning `{data, error, isLoading, isValidating, loadedSome, active, …}` mirroring exactly what the component's hook call returns (`active: false` when the key expression is currently `null`) — so property predicates read the same projection the application code reads instead of raw template variables. The template is part of the trusted base and gets its own differential tests against real SWR behavior (Spec 04 §6).

**Multi-parameter keys and the key window.** A key class spanning several abstract variables (the checkout walkthrough's `['quote', uid, plan, seats, cycle, coupon]`) would instantiate the template once per parameter combination — dozens of entries, a state-space multiplier with no property-relevant payoff. Instead the template keeps **full entries only for a bounded window of recently-current keys** (default W=2: current + previous); entries evicted from the window collapse into a summary, and a key re-entering after eviction gets a **havoc'd entry** (data/error unconstrained — sound over-approximation) while key-change revalidation still fires. Two semantics are load-bearing and normative for every cache-shaped template:

1. **Per-key isolation** — a resolve writes only to its op's key entry, never to "the current view"; and
2. **the view reads the current key only**.

Staleness properties ("an old response must not become the valid quote for the current cart") hold or fail through these two rules, so they are *template invariants* — established once by the template's differential tests, not re-derived per app. Window evictions are bound-hit events in the trust ledger.
